# 第三方登录服务集成指南

## 概述

本系统支持通过配置的方式集成第三方医院的登录API。系统会根据用户输入的医院名称，自动路由到对应的第三方登录接口进行身份验证。

## 工作流程

1. 用户在登录页面输入医院名称、用户名和密码
2. 系统判断医院名称是否为"系统"
   - 如果是"系统"，使用本地数据库验证
   - 如果是其他医院名称，使用第三方登录服务
3. 第三方登录服务查找医院配置和服务配置
4. 调用第三方API进行验证
5. 返回验证结果和令牌

## 配置步骤

### 1. 添加医院基本信息

在管理后台的"医院基本信息配置"中添加医院：

- **医院名称**: 例如 "某某医院"
- **医院编码**: 例如 "HOSPITAL001"
- **其他信息**: 地址、联系方式等（可选）

### 2. 配置第三方登录服务

在管理后台的"医院服务配置"中为该医院添加登录服务配置：

#### 基本配置

- **医院**: 选择前面创建的医院
- **服务名称**: 例如 "第三方登录服务"
- **服务类别**: 选择 "认证服务"
- **服务发现键值**: **必须填写** `LoginSystem`（固定值，与API接口路径对应）
- **服务地址**: 第三方API的基础地址，例如 `https://api.hospital.com`
- **服务端口**: 如果与地址中的端口不同，可以单独指定（可选）
- **API版本**: 例如 `v1`（可选）

#### 认证配置（可选）

如果第三方API需要认证：

- **认证类型**: 
  - `ApiKey`: 使用API密钥认证
  - `Basic`: 使用Basic认证
- **API密钥**: 如果选择ApiKey认证，填写密钥
- **用户名/密码**: 如果选择Basic认证，填写用户名和密码

#### 高级配置

- **超时时间(秒)**: 请求超时时间，默认30秒
- **重试次数**: 失败后重试次数，默认0次
- **健康检查地址**: 用于测试服务可用性的URL（可选）
- **是否启用**: 必须勾选，否则服务不会被调用
- **优先级**: 如果同一医院有多个登录服务配置，优先使用优先级高的

## API接口规范

第三方登录API必须符合以下规范：

### 请求

- **方法**: POST
- **路径**: `/api/auth/LoginSystem`
- **Content-Type**: `application/json`

#### 请求体

```json
{
  "UserName": "用户名",
  "Password": "密码"
}
```

### 响应

#### 成功响应 (HTTP 200)

```json
{
  "AccessToken": "访问令牌字符串",
  "RefreshToken": "刷新令牌字符串"
}
```

#### 失败响应

返回非200状态码，系统会记录错误日志并提示用户登录失败。

## URL构建规则

系统会按照以下规则构建完整的API URL：

```
基础地址/API版本/api/auth/LoginSystem
```

示例：

- 基础地址: `https://api.hospital.com`
- API版本: `v1`
- 端口: `8080`
- 最终URL: `https://api.hospital.com:8080/v1/api/auth/LoginSystem`

## 使用示例

### 配置示例

假设要集成"XX市人民医院"的登录系统：

1. **医院基本信息**
   - 医院名称: `XX市人民医院`
   - 医院编码: `XXSRMYY`

2. **服务配置**
   - 服务名称: `XX市人民医院登录服务`
   - 服务类别: `认证服务`
   - 服务发现键值: `LoginSystem`
   - 服务地址: `https://his.xxhospital.com`
   - 服务端口: `443`
   - API版本: `v2`
   - 认证类型: `ApiKey`
   - API密钥: `your-api-key-here`
   - 超时时间: `30`
   - 重试次数: `2`
   - 是否启用: `是`

### 登录流程

1. 用户在登录页面输入：
   - 医院名称: `XX市人民医院`
   - 用户名: `doctor001`
   - 密码: `password123`

2. 系统自动：
   - 查找"XX市人民医院"的配置
   - 找到LoginSystem服务配置
   - 构建URL: `https://his.xxhospital.com:443/v2/api/auth/LoginSystem`
   - 添加API-Key请求头
   - 发送POST请求
   - 接收并解析响应

3. 验证成功后：
   - 创建本地登录会话
   - 保存用户信息到Claims
   - AccessToken和RefreshToken可用于后续API调用
   - 重定向到首页

## 故障排查

### 登录失败

1. **检查医院配置是否存在**
   - 确认医院名称输入正确（区分大小写）
   - 在管理后台查看医院是否已添加

2. **检查服务配置**
   - 确认服务发现键值为 `LoginSystem`
   - 确认服务状态为"启用"
   - 检查服务地址是否正确

3. **查看日志**
   - 检查应用日志中的错误信息
   - 确认第三方API返回的错误详情

4. **测试连接**
   - 使用Postman等工具直接测试第三方API
   - 确认网络连接正常
   - 验证认证信息是否正确

### 常见错误

- **"医院配置不存在"**: 医院基本信息未配置或医院名称不匹配
- **"未配置第三方登录服务"**: 该医院的LoginSystem服务配置不存在或未启用
- **"第三方登录失败，状态码: XXX"**: 第三方API返回错误，检查认证信息和请求参数
- **"第三方登录响应格式错误"**: API响应不符合规范，缺少AccessToken字段
- **"登录服务异常"**: 网络连接失败或超时

## 安全建议

1. **使用HTTPS**: 所有第三方API地址应使用HTTPS协议
2. **密钥管理**: API密钥和密码应妥善保管，定期更换
3. **超时设置**: 合理设置超时时间，避免请求hang住
4. **日志审计**: 定期检查登录日志，发现异常及时处理
5. **令牌存储**: 建议将AccessToken和RefreshToken加密存储

## 扩展功能

### 令牌刷新

如果需要使用RefreshToken刷新AccessToken，可以：

1. 在Claims中存储RefreshToken
2. 创建专门的令牌刷新服务
3. 在令牌即将过期时自动刷新

### 多服务集成

同一医院可以配置多个服务（如报告查询、检查预约等），通过不同的服务发现键值区分：

- `LoginSystem`: 登录服务
- `ReportQuery`: 报告查询服务
- `AppointmentService`: 预约服务

## 技术细节

### 相关文件

- **模型**: `Models/ThirdPartyLoginRequest.cs`
- **服务接口**: `Services/IThirdPartyLoginService.cs`
- **服务实现**: `Services/ThirdPartyLoginService.cs`
- **登录控制器**: `Pages/Login.cshtml.cs`
- **服务注册**: `Program.cs`

### 依赖服务

- `IHospitalProfileService`: 医院基本信息服务
- `IHospitalServiceConfigService`: 医院服务配置服务
- `IHttpClientFactory`: HTTP客户端工厂

### 重试机制

系统支持自动重试失败的请求：

- 重试次数可配置（0-10次）
- 每次重试间隔递增（1秒、2秒、3秒...）
- 只有网络异常或非成功状态码才会重试
